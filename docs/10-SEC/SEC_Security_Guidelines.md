## 1. 安全编码规范

### 1.1 输入验证

- 所有用户输入必须经过严格验证，拒绝任何未预料的格式和数据
- 白名单验证优先于黑名单验证
- 服务端验证是最后防线，客户端验证仅为 UX 优化

```
// 推荐：明确允许的字符集
const safeFilename = input.replace(/[^a-zA-Z0-9._-]/g, '');

// 禁止：仅移除危险字符（可被绕过）
const unsafe = input.replace(/flag/g, '');  // 不安全
```

### 1.2 输出编码

- 根据输出上下文（HTML、JS、CSS、URL、SQL）使用正确的编码方式
- 对任何用户可控的数据进行转义，防止 XSS

| 上下文 | 编码类型 | 示例 |
|--------|----------|------|
| HTML | HTML实体编码 | `&lt;` `&gt;` `&amp;` |
| JavaScript | JSON / 转义 | `JSON.stringify()` |
| URL | URL编码 | `encodeURIComponent()` |
| SQL | 参数化查询 | 预编译语句 |

### 1.3 认证与会话管理

- 使用业界标准认证协议（OAuth 2.0、OIDC、SAML）
- 会话 ID 必须足够随机，长度 ≥ 128 bit，使用安全随机数生成器
- 会话 cookie 必须设置：`Secure`、`HttpOnly`、`SameSite=Strict`
- 会话超时：idle 超过 30 分钟自动失效
- 禁止在 URL、日志、错误消息中包含会话 ID

### 1.4 访问控制

- 默认拒绝（deny-by-default）
- 遵循最小权限原则，仅授予完成任务所需的最小权限
- 服务之间调用使用 mTLS 或 API Key + 签名进行双向认证
- 敏感操作必须记录审计日志（who/what/when/result）

### 1.5 依赖管理

- 禁止使用已知的 CVE 漏洞库，依赖更新窗口不超过 30 天
- 定期运行 `npm audit` / `cargo audit` / `owasp dependency-check`
- 固定第三方依赖的精确版本并锁定 hash（package-lock.json, Gosum）

### 1.6 日志与错误处理

- 禁止记录：密码、密钥、token、完整 PII、信用卡数据
- 生产环境禁止输出详细堆栈信息，使用通用错误消息 + 错误 ID
- 日志必须包含：时间戳（UTC）、请求 ID、用户 ID（若已认证）、操作类型、结果

## 2. 加密规范

### 2.1 对称加密

| 算法 | 密钥长度 | 备注 |
|------|----------|------|
| AES-128-GCM | 128 bit | 首选 |
| AES-256-GCM | 256 bit | 高敏感数据 |

- 禁止使用：DES、3DES、ECB 模式、AES-ECB
- IV 必须使用 `crypto.randomBytes()` 生成，每次加密必须唯一
- 使用 AEAD 模式（AES-GCM）时 auth tag 长度为 128 bit

### 2.2 非对称加密

| 用途 | 算法 | 最小密钥长度 |
|------|------|--------------|
| 密钥交换 | ECDH | P-256 或更高 |
| 数字签名 | ECDSA | P-256 或更高 |
| 通用加密 | RSA-OAEP | 2048 bit（2030年前）/ 3072 bit |

- 禁止使用：RSA-PKCS1v1.5（易被攻击）、1024 bit 以下 RSA
- 签名算法优先使用 ECDSA（P-256），性能优于 RSA

### 2.3 哈希函数

- 用于完整性校验：SHA-256 或更强（SHA-384、SHA-512）
- 禁止使用：MD5、SHA-1（除兼容性场景）、MD4
- 口令哈希必须使用 bcrypt、Argon2 或 PBKDF2（迭代 ≥ 10,000）

### 2.4 密钥管理

- 密钥不得硬编码在代码中，必须存储在 Vault（HashiCorp Vault、AWS KMS）
- 密钥轮换周期不超过 1 年，轮换期间支持双密钥并行验证
- 密钥必须以加密形式存储，静态密钥（at-rest）使用 AES-256 加密
- 密钥泄露后立即轮换，所有使用该密钥加密的数据需重新评估

### 2.5 随机数生成

- 必须使用加密安全的随机数生成器（CSPRNG）
- Python: `secrets` 模块；Node.js: `crypto.randomBytes()`；Go: `crypto/rand`

## 3. 通信安全

### 3.1 传输加密

- 所有网络通信必须使用 TLS 1.2+，禁用 TLS 1.0/1.1 及早期版本
- TLS 配置遵循 Mozilla Server Side TLS Modern Compatibility 配置
- 证书必须由可信 CA 签发（禁止自签名证书用于生产环境）
- 证书链必须完整，中间证书必须部署

### 3.2 证书管理

- 证书有效期不超过 13 个月，推荐 ACME 自动续期
- 证书吊销检查（CRL/OCSP）应在 24 小时内完成
- 使用 HSTS（Strict-Transport-Security）头部，最小 max-age = 6 个月
- 包含 `preload` 标志并提交至 HSTS Preload List

### 3.3 API 安全

- 强制使用 HTTPS，不接受 HTTP 请求
- API 版本控制在 URL 或 Header 中明确指定
- 所有 API 必须有认证与授权验证，使用 OAuth 2.0 / JWT
- JWT 必须校验 `exp`、`iat`、`iss`、`aud`，禁止使用 `alg: none`
- 请求频率限制（Rate Limiting）：每 IP 每分钟 ≤ 1000 请求

### 3.4 DNS 安全

- 关键域名启用 DNSSEC
- 使用 ECS（EDNS Client Subnet）时注意隐私权衡

## 4. 数据存储

### 4.1 数据分类

| 等级 | 定义 | 加密要求 |
|------|------|----------|
| 公开 | 可对外披露 | 不需要 |
| 内部 | 仅内部员工访问 | 静态加密（AES-256） |
| 机密 | 限制范围访问 | 静态加密 + 传输加密 |
| 绝密 | 最高敏感数据 | 静态加密 + 专用密钥管理 + 审计 |

### 4.2 静态加密（At-Rest）

- 数据库字段级加密用于敏感数据（信用卡、证件号、密钥）
- 存储系统（S3/OSS/GCS）必须启用服务端加密（SSE-S256 或 SSE-KMS）
- 备份数据必须与主数据同等加密Level
- 加密密钥与加密数据不得存储在同一系统

### 4.3 数据库安全

- 最小化数据库账户权限，禁止应用使用 root/admin 账户
- 禁用网络直连，仅允许通过代理（PgBouncer、连接池）
- 敏感字段使用应用层加密，而非依赖 DBMS 透明加密
- 定期轮换数据库密码，周期 ≤ 90 天

### 4.4 文件存储

- 上传文件必须验证类型（魔数检测，不仅依赖扩展名）
- 文件大小限制：单文件 ≤ 100 MB
- 禁止在 web 根目录或可访问路径存储敏感文件
- 临时文件使用后立即删除，禁止留在文件系统

### 4.5 数据删除

- 删除数据时执行安全擦除（覆写）或密钥销毁
- 缓存数据 TTL 最大值：非敏感 24 小时，敏感 1 小时

## 5. 隐私保护

### 5.1 数据收集最小化

- 仅收集业务必需的最少个人数据
- 收集前必须获得明确、具体、知情的用户同意
- 隐私政策必须使用通俗语言，版本变更需重新通知用户

### 5.2 PII 处理

- PII（个人身份信息）包括但不限于：姓名、手机号、身份证、邮箱、IP、设备 ID
- PII 处理必须有明确法律依据（合同履行、法律义务、用户同意）
- 禁止在测试/开发环境使用真实生产数据，必须使用脱敏数据
- 数据脱敏规则：手机号中间4位掩码、邮箱 @ 后缀域名保留

### 5.3 数据主体权利

支持并实施以下权利：

- 知情权：提供数据收集的透明说明
- 访问权：用户提供身份验证后可查看其个人数据
- 更正权：用户可请求更正不准确数据
- 删除权：用户可请求删除数据（有合法保留理由的除外）
- 可携带权：提供结构化、机器可读的数据导出（JSON/CSV）

响应时限：收到请求后 **72 小时内** 确认收悉，**30 天内** 完成处理

### 5.4 数据保留

- 制定数据保留策略，明确每类数据的保留期限
- 超过保留期限的数据必须删除或匿名化处理
- 保留日志（who/when/what）至少 1 年

### 5.5 跨境数据传输

- 个人数据跨国传输必须符合当地数据保护法规（如 GDPR、中国《数据安全法》）
- 使用标准合同条款（SCC）、充分性认定或 Binding Corporate Rules
- 记录所有跨境传输活动并保存证据

## 6. 应急响应

### 6.1 事件分级

| 级别 | 定义 | 响应时限 |
|------|------|----------|
| P0 — 紧急 | 正在发生或即将发生的数据泄露、系统瘫痪 | 立即（15分钟内） |
| P1 — 高 | 已确认安全事件，影响核心业务 | 1 小时内 |
| P2 — 中 | 疑似安全事件或低风险漏洞 | 4 小时内 |
| P3 — 低 | 潜在风险、观察到异常 | 24 小时内 |

### 6.2 响应流程

```
发现 → 评估 → 遏制 → 消除 → 恢复 → 事后分析
  ↓
记录时间 + 指派负责人
```

### 6.3 关键联系人

| 角色 |职责 | 联系 |
|------|------|------|
| 安全负责人 | 决策、协调 | [由安全团队维护] |
| 技术应急团队 | 技术遏制与恢复 | [由安全团队维护] |
| 法务/合规 | 监管报告、通知义务 | [由安全团队维护] |
| 公关/沟通 | 外部沟通、媒体应对 | [由安全团队维护] |

> 完整联系人列表存储于：[安全团队 Wiki / 内部应急手册]

### 6.4 遏制措施（快速止血）

- 账户泄露：立即吊销 token/session，强制重置密码
- 服务器沦陷：立即隔离（网络ACL/防火墙），不得直接远程修复
- 数据泄露：隔离存储系统、启动日志留存、评估是否需要下线
- 勒索软件：禁止自行支付，立即联系安全团队

### 6.5 监管报告

- 个人信息泄露：知道后 **72 小时内** 向监管机构报告（如适用）
- 重大数据泄露可能需要通知受影响用户（根据适用法律）
- 所有安全事件必须记录，保存至少 3 年

### 6.6 事后分析（Post-Incident Review）

每个 P0/P1 事件结束后 5 个工作日内完成：

1. 事件时间线（精确到分钟）
2. 根本原因分析（Root Cause Analysis）
3. 影响范围评估
4. 应对过程评估（what went well / what didn't）
5. 改进行动项（Action Items）+ 责任人 + 完成日期
6. 更新应急响应预案

## 7. 合规与审计

- 每年至少进行一次第三方安全评估（渗透测试）
- 安全控制必须可审计：记录足够日志支持事后调查
- 关键基础设施变更必须经过安全评审（Security Review）
- 违反本规范的代码/配置变更必须阻断并上报
